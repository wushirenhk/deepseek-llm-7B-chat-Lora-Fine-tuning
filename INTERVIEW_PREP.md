## ⚠️ 技术难点与解决方案（高频问题）

### 难点 1：多轮对话如何序列化？

**问题**：原始数据是多轮对话列表，模型只能接受单个连续序列，如何处理？

**面试官想考察**：你对数据处理的理解，是否能处理复杂的对话格式

**回答要点**：

```python
# 原始数据格式
conversation = [
    {"role": "user", "content": "我最近焦虑..."},
    {"role": "assistant", "content": "我理解你的感受..."},
    {"role": "user", "content": "我该怎么办..."},
    {"role": "assistant", "content": "建议你尝试..."}
]

# 转换后的模板格式
"<|bos|>system<|user|>我最近焦虑...<|assistant|>我理解你的感受...<|user|>我该怎么办...<|assistant|>建议你尝试...<|eos|>"
```

**关键点**：

- 按顺序遍历每轮对话，用特殊 token (`<|user|>` `<|assistant|>`) 分隔
- 使用模型自带的 `bos_token` 和 `eos_token`
- 每条数据是**一个完整的对话历史**，而不是单轮 QA

> **面试背诵段落**：
> "多轮对话的序列化核心是**用特殊token标记角色**。我将对话列表按顺序拼接，用`<|user|>`和`<|assistant|>`标记每轮发言的角色，在助手回复前后分别加上bos和eos token。这样模型看到的就是一个完整的对话上下文，能学习如何根据历史生成下一个回复。关键是不拆分多轮，每条数据都是独立完整的对话。"

### 难点 2：Label Masking（标签掩码）

**问题**：多轮对话中，哪些部分需要计算 loss、哪些需要忽略？

**面试官想考察**：你对语言模型训练机制的理解

**回答要点**：

```python
# 核心代码
labels = (
    [-100] * len(input_ids) +      # 用户输入 → 忽略，不计算梯度
    output_tokenizer["input_ids"] + # 助手回复 → 计算梯度
    [tokenizer.eos_token_id]
)
```

**为什么这样做？**

1. **只学习"回答"**：大模型本质是"续写"，我们用 `<|assistant|>` 开头，让模型学习生成专业回复
2. **避免学习"问题"**：用户输入是"问题"不是"答案"，不应该让模型模仿用户说话
3. **提高训练效率**：减少约 50% 的梯度计算量

**图示**：

```
┌────────────────────────┬──────────────────────────────────┐
│      User Input         │        Assistant Output          │
│   (不该学习这个)         │      (正是要学习的)              │
├────────────────────────┼──────────────────────────────────┤
│   Labels: -100          │   Labels: [actual token ids]    │
│   (loss = 0)            │   (loss = cross_entropy)        │
└────────────────────────┴──────────────────────────────────┘
```

> **面试背诵段落**：
> "Label Masking 是为了让模型**只学习怎么回答，不学习怎么提问**。我把用户输入部分的label设为-100，这样loss=0不计算梯度；只保留助手回复部分的loss让模型学习。原因是大模型本质是续写，我们用`<|assistant|>`开头让模型生成专业回答。如果让模型同时学习用户输入，会降低训练效率甚至让模型学会模仿用户提问，这是我们不希望的。"

### 难点 3：LoRA 原理与参数选择

**问题**：LoRA 的核心原理是什么？为什么选择 rank=64？

**面试官想考察**：你对 LoRA 底层原理的理解，而非只会调库

**回答要点（画图解释）：**

```
原始全连接层:  W₀ ∈ ℝ^(d×k)

LoRA 核心思想：W = W₀ + ΔW = W₀ + BA
                        其中 B ∈ ℝ^(d×r), A ∈ ℝ^(r×k), r << d

冻结 W₀，只训练 A 和 B
```

**为什么这样设计？**

1. **理论支撑**：神经网络权重通常有低的本征秩 (intrinsic rank)，用低秩分解可以有效学习新任务
2. **效率提升**：从 d×k 参数量降到 r×(d+k)，通常 r=8/16/64，远小于 d
3. **可插拔**：多个任务训练多个 LoRA adapter，推理时动态加载

**rank=64 的选择依据：**

| rank         | 参数量 | 效果               | 适用场景              |
| ------------ | ------ | ------------------ | --------------------- |
| 8/16         | ~1M    | 基础能力适配       | 快速实验              |
| 32           | ~4M    | 中等效果           | 资源受限              |
| **64** | ~7M    | **最佳效果** | **生产环境** ✅ |
| 128          | ~14M   | 边际提升小         | 特殊需求              |

**回答**：rank=64 是在效果和效率之间的平衡点。实际测试中，rank 超过 64 后提升有限，但训练时间和显存会继续增加。

> **面试背诵段落**：
> "LoRA核心是**低秩分解**。把原始权重W₀冻结，新增两个矩阵A和B，前向时W=W₀+BA，训练只更新AB。为什么有效？论文证明神经网络权重有低本征秩，新任务知识可以压缩到低秩空间。rank=64是因为它是效果和效率的平衡点——rank太小学不到足够知识，rank太大提升有限但显存翻倍。实际测试rank 64以上边际收益递减。"

### 难点 4：显存优化

**问题**：7B 模型如何能在单卡 16GB 显存下训练？

**面试官想考察**：工程落地能力，显存优化的实际经验

**回答要点（显存优化组合拳）：**

```python
# 1. 半精度训练 - 显存减半
model = AutoModelForCausalLM.from_pretrained(
    model_path,
    torch_dtype=torch.float16  # float32 → float16，显存减半
)

# 2. 梯度检查点 - 显存再降 30%
train_args = TrainingArguments(
    gradient_checkpointing=True  # 用计算换显存
)

# 3. 梯度累积 - 等效大 batch
train_args = TrainingArguments(
    per_device_train_batch_size=2,      # 实际 batch
    gradient_accumulation_steps=8,      # 累积 8 步 = 等效 batch=16
)

# 4. 智能设备分配
model_kwargs = {"device_map": "auto"}   # 自动分配到多卡或单卡
```

**显存计算**：

```
7B 参数 × float16 = 14GB 原始模型
+ 梯度 (14GB) × float16 = 14GB
+ 优化器状态 (Adam: 2×参数) = 28GB
+ 激活值 (与 batch 有关)

总计约 28-40GB（全量微调）

使用 LoRA 后：
冻结 W₀ → 不保存梯度 → -14GB
只训练 LoRA 参数 → +0.01GB

总计约 16GB ✅（单卡可跑）
```

> **面试背诵段落**：
> "7B模型单卡16GB能跑，核心是**四招组合**：第一，**半精度fp16**让模型权重从28GB降到14GB；第二，**冻结预训练权重**不保存梯度省14GB；第三，**梯度检查点**用计算换显存再省30%；第四，**梯度累积**用小batch分步累积达到大batch效果。这四招叠加，LoRA训练只需16GB，单卡3090就能跑。"

### 难点 5：LoRA 目标模块选择

**问题**：为什么针对 `q_proj, k_proj, v_proj, o_proj` 等这些层？

**面试官想考察**：对 Transformer 架构的理解深度

**回答要点（画 Attention 结构）：**

```
        ┌─────────────────────────────────────┐
Input ──┤ q_proj (Query 投影)                  │
        │ k_proj (Key 投影)                    │──▶ Attention 计算
        │ v_proj (Value 投影)                 │    Q @ K^T @ V
        │ o_proj (Output 投影)                 │
        └─────────────────────────────────────┘

LoRA 的 insight：
这些投影矩阵是 Transformer "理解语义" 的核心
修改它们 = 调整模型的 "理解方式"
```

**本项目选择的模块**：

```python
target_modules = [
    'up_proj',   # FFN 升维 (experts 概念相关)
    'gate_proj', # Gate 门控
    'q_proj',    # Query 投影 ← 核心
    'o_proj',    # Output 投影 ← 核心
    'down_proj', # FFN 降维
    'v_proj',    # Value 投影 ← 核心
    'k_proj',    # Key 投影 ← 核心
]
```

**为什么不加 `lm_head`？** 因为 `lm_head` 是最终分类层，学习它会破坏预训练语言能力。

> **面试背诵段落**：
> "LoRA通常加在**Q、K、V、O四个投影矩阵**上，因为这些是Transformer'理解语义'的核心。Attention机制是Q查K索引V内容，O做输出投影，修改这些层就是调整模型的'理解方式'。Q/K负责语义匹配，V/O负责内容表达。项目中我还加了FFN的up_proj、gate_proj、down_proj，因为DeepSeek的FFN也有可学习空间。不加lm_head是因为它是最终分类层，学它会破坏预训练语言能力。"

### 难点 6：模型权重合并

**问题**：为什么要合并 LoRA 权重？合并后模型还能用吗？

**面试官想考察**：对 LoRA 部署方式的理解

**回答要点：**

```
LoRA 训练时：
┌─────────────┐     ┌─────────────┐
│  Base Model │  +  │ LoRA Adapter │  ──▶  加载时需要分别加载
└─────────────┘     └─────────────┘

合并后：
┌─────────────────────────┐
│    Merged Model         │  ──▶  单独一个模型，可直接加载推理
└─────────────────────────┘
```

**合并原理**：

```python
# PEFT 提供的合并方法
merged_model = model.merge_and_unload()
# 等价于: W_merged = W_base + W_lora
```

**为什么合并？**

1. **部署简单**：不需要加载 base model + adapter 两部分
2. **推理速度更快**：一次加载，少一次 adapter 融合开销
3. **兼容性更好**：某些推理框架不支持动态加载 adapter

> **面试背诵段落**：
> "合并LoRA权重主要是为了**简化部署和提升推理速度**。训练时模型是base model加adapter两部分，合并后W_merged=W_base+W_lora，变成单一完整模型。好处是：一不用加载两份模型文件，二推理框架兼容性更好，三推理时少一次adapter融合开销。合并用PEFT的merge_and_unload()一键完成，非常方便。"

### 第二类：技术原理类

---

**Q7: LoRA 的核心原理是什么？**

> "LoRA核心是**低秩分解**W=W₀+BA。冻结预训练权重W₀，只训练AB两个低秩矩阵，其中r远小于d和k。理论依据是预训练权重有低本征秩，新任务知识可压缩到低秩空间。7B模型LoRA只需训练约0.1%参数量（7M参数），显存从28GB降到16GB，但效果接近全量微调。"

---

**Q8: LoRA 和全量微调的区别？**

> "核心区别有四点：**参数量**：全量微调更新所有参数（7B），LoRA只更新0.1%（7M）；**显存**：全量微调需要28GB+，LoRA只需16GB；**效果**：LoRA效果可达全量微调的95-99%；**适用场景**：全量微调适合数据量大、任务复杂的场景，LoRA适合数据量有限、资源受限的生产环境。LoRA的优势是训练快、显存低、可快速切换任务。"

---

**Q9: LoRA 的 rank 如何选择？**

> "Rank选择看效果和效率的平衡。**rank 8/16**：约1M参数，适合快速实验；**rank 32**：约4M参数，资源受限可用；**rank 64**：约7M参数，效果最佳，是生产环境常见选择；**rank 128**：约14M参数，边际提升小。一般推荐从64开始尝试，如果效果不好或数据量特别大可增加到128，但超过64后提升通常有限。"

---

**Q10: 为什么要用半精度 (fp16/bf16) 训练？**

> "半精度训练主要是**节省显存**。float32占4字节，float16占2字节，同样7B参数模型，fp16只需要14GB而fp32需要28GB。精度损失通常可接受，因为预训练模型本身就有一定冗余度，BERT、GPT等模型的实验都证明fp16训练效果和fp32相当。现代NVIDIA GPU还有tensor core加速fp16/bf16运算，训练速度也更快。"

---

**Q11: 梯度检查点 (gradient_checkpointing) 原理？**

> "梯度检查点是**用计算换显存**。普通训练会保存所有层的激活值用于反向传播，占用大量显存。开启检查点后只保存部分激活值，反向传播时需要重新计算中间层。代价是训练时间增加20-30%，但显存节省50-70%。开启方式是一行代码`model.gradient_checkpointing_enable()`。"

---

**Q12: 梯度累积 (gradient_accumulation) 解决什么问题？**

> "梯度累积解决**显存不足时的大batch训练问题**。比如单卡只能放batch=2，但实际需要batch=32。做法是每次前向传播计算梯度但不更新参数，累积8次后统一更新，等效达到batch=16效果。实现方式是设置`gradient_accumulation_steps=8`，代码上就是循环执行前向和累积，最后统一反向更新。"

---

**Q13: Attention 的 Q/K/V 各自作用？**

> "Q（Query）是查询向量，K（Key）是索引向量，V（Value）是内容向量。Attention计算是**Q @ K^T**得到匹配分数，再用softmax归一化，最后乘以V得到加权求和。通俗理解：Q是'我想查询什么'，K是'我有哪些关键词'，V是'这些关键词对应什么内容'。Q和K决定注意力权重，V决定传递的信息内容。"

---

**Q14: 为什么 LoRA 通常只加在 Q/V 层？**

> "经验发现Q和V是Transformer**理解语义的核心**。Q负责语义匹配，决定关注什么；V负责内容表达，决定传递什么信息。修改它们能最有效地调整模型的"理解方式"。实践中通常加在Q、K、V、O四个投影矩阵上，这四个是Attention的核心组件。实验中发现，加这些层比加FFN或其他层效果更好。"

---

**Q15: 大模型为什么需要微调？**

> "因为**通用大模型缺乏领域知识**。预训练模型学习的是通用文本分布，对医疗、法律等专业领域理解不深。比如医疗问答需要专业术语和诊断逻辑，通用模型可能给出似是而非的回答。微调通过在领域数据上训练，让模型学会领域特定的表达方式、推理逻辑和专业术语，从而提供更准确、更专业的回答。"

---

**Q16: LoRA 训练时为什么冻结 base model？**

> "冻结base model主要有两个原因：**保留预训练知识**和**避免灾难性遗忘**。如果更新所有参数，新任务的数据分布可能覆盖预训练学到的通用知识，导致模型丧失原有能力。冻结W₀后，只在LoRA参数中学习新任务，预训练知识得以保留。此外冻结也节省显存——不保存冻结部分的梯度，只需要约原来5%的显存即可训练。"

---

**Q17: 什么是灾难性遗忘？如何缓解？**

> "灾难性遗忘是神经网络**学新任务时突然忘记旧知识**。全量微调时所有参数都被更新，新任务数据可能覆盖预训练知识。LoRA通过**冻结预训练权重**来缓解——W₀固定不变，只在LoRA参数中学新任务，对预训练知识影响最小。低秩更新也意味着参数量少，破坏预训练知识的程度也低。"

---

### 第三类：数据处理类

---

**Q18: 数据格式是什么样的？**

> "原始数据是**JSON格式**，每个样本是一个多轮对话数组。每个对话包含多个消息，每条消息有role字段（user/assistant/system）和content字段。比如：'[{"role":"user","content":"我最近焦虑..."},{"role":"assistant","content":"我理解你的感受..."}]'，表示一轮完整的问答。"

---

**Q19: 如何处理多轮对话？**

> "多轮对话处理分三步：**拼接+特殊token分隔+Label Masking**。首先按顺序拼接所有消息，用`<|user|>`和`<|assistant|>`标记角色，在开头加bos token，结尾加eos token。然后把拼接后的文本送给模型进行训练。关键是**不拆分每轮对话**，每条训练样本都是完整的多轮上下文，让模型学习根据完整历史生成回复。"

---

**Q20: 对话数据如何转换为训练样本？**

> "首先用模型自带的**对话模板**格式化。我用的DeepSeek模型对话格式是：'<|bos|>system{system_prompt}<|user|>user_input<|assistant|>assistant_output<|eos|>'。然后tokenizer处理成token ids。注意多轮对话是把所有历史消息拼接在一起，而不是只保留最后一轮，这样才能让模型学习上下文推理能力。"

---

**Q21: Label Masking 具体怎么做？**

> "核心是把**用户输入部分的label设为-100**，这样loss=0不计算梯度，只保留助手回复部分的loss。具体代码是：labels = [-100] * len(input_ids) + assistant_input_ids + [eos_token_id]。用户输入是"问题"不是"答案"，不应该让模型模仿用户说话。大模型本质是续写，我们用`<|assistant|>`开头让模型生成专业回复。"

---

**Q22: 数据量多少？够不够？**

> "项目有**几千条**高质量医疗对话。对于LoRA微调，这个量级是够的。因为LoRA只训练0.1%参数，参数量少所以不容易过拟合，对数据量需求比全量微调小很多。当然数据越多越好，但高质量医疗对话标注成本高。实践中几千条专业标注数据已经能让LoRA达到不错的效果。全量微调通常需要至少10万条数据才够。"

---

**Q23: 数据如何清洗？**

> "数据清洗主要包括：**去重**——删除重复或高度相似的对话，避免训练多样性下降；**过滤无效对话**——删除过短、内容乱码、格式错误的样本；**标准化格式**——统一角色名称、特殊符号等；**质量审核**——抽检确保医患对话的专业性和准确性。清洗后的数据按8:1:1分成训练集、验证集、测试集。"

---

### 第四类：模型训练类

---

**Q24: 训练参数如何设置？**

> "核心参数：**batch_size=2**（per_device），**gradient_accumulation_steps=8**（等效batch=16），**learning_rate=2e-5**，**num_train_epochs=3**，**warmup_ratio=0.1**。其他重要参数：weight_decay=0.01防止过拟合，logging_steps=10监控训练，save_steps=100保存checkpoint。LoRA参数：rank=64，lora_alpha=32，target_modules包含QKV和FFN层。"

---

**Q25: 学习率为什么设置 2e-5？**

> "学习率2e-5是**预训练模型微调的经验值**。预训练模型已经具备强大的语言能力，过大的学习率会破坏已学到的知识，导致灾难性遗忘。过小的学习率又训练太慢。通常预训练用1e-4到1e-5，而微调因为要保护预训练知识，会用更小的1e-5到5e-5。LoRA因为只训练少量参数，可以适当放宽，但一般还是用2e-5左右。"

---

**Q26: 训练多少个 epoch？**

> "一般训练**3-5个epoch**。LoRA参数量少，不容易过拟合，但也不是训练越多越好。训练太多反而可能导致LoRA权重过大，对预训练知识的干扰增加。实践中会监控验证集loss，如果连续2-3个epoch验证集loss不下降就提前停止。3个epoch是常见的安全选择。"

---

**Q27: 如何判断模型训练好了？**

> "判断训练好坏看**两个指标**：训练loss下降曲线和验证集指标。训练loss持续下降说明模型在学习；验证集loss不再下降甚至上升说明可能过拟合。同时监控生成文本质量——抽几个测试prompt看模型输出是否流畅、专业。医疗场景还会做人工评估。最终上线前在测试集上跑完整评估流程。"

---

**Q28: 训练过程中遇到过什么问题？**

> "主要遇到两类问题：**显存溢出**和**loss不收敛**。显存溢出通过减小batch_size、增加gradient_accumulation_steps解决。Loss不收敛可能是学习率不当或数据质量问题，尝试调低学习率或检查数据格式。NaN问题通常是梯度爆炸，加gradient_clipping解决。LoRA训练一般比较稳定，这些问题发生频率比全量微调低很多。"

---

**Q29: 如何避免过拟合？**

> "防止过拟合常用几招：**weight_decay**（L2正则化）防止权重过大；**dropout**在LoRA层加少量dropout；**early stopping**监控验证集loss提前停止；**数据增强**增加训练多样性。LoRA本身因为参数量少就不容易过拟合，再加上冻结base model约束，过拟合风险比全量微调小很多。"

---

**Q30: 模型训练用了多长时间？**

> "在**单卡A100 80GB**上，大约需要X小时完成3个epoch。实际时间取决于batch_size、序列长度、gradient_accumulation_steps等因素。使用LoRA比全量微调快很多，因为可训练参数从7B降到7M。如果用多卡并行还能进一步加速，DeepSpeed ZeRO-3可以把训练时间缩短到原来的三分之一。"

---

### 第五类：模型部署类

---

**Q31: 如何部署微调后的模型？**

> "部署流程是：**合并LoRA → 保存完整模型 → 加载推理**。训练完的模型是base model加LoRA adapter两部分，需要先合并成单一模型。用PEFT的`model.merge_and_unload()`一键合并，然后`model.save_pretrained()`保存。部署时直接用Transformers加载合并后的完整模型，可以像加载普通预训练模型一样使用。"

---

**Q32: LoRA adapter 可以单独加载吗？**

> "可以。PEFT支持**动态加载adapter**，不需要合并。推理时用`PeftModel.from_pretrained(model, adapter_path)`加载指定adapter，加载后base model会自动加上LoRA权重。这种方式的好处是可以**同时加载多个adapter**——比如医疗、法律、客服三个领域各训练一个adapter，推理时动态切换。模型文件也小很多，只保存几十MB的adapter而不是14GB完整模型。"

---

**Q33: 推理速度如何？**

> "推理速度取决于硬件和batch配置。7B模型在**A100上约30-50 tokens/s**，在3090上约20-30 tokens/s。批量推理可以通过增大batch提升吞吐量，但会增加延迟。优化手段包括：KV cache缓存已计算的key/value避免重复计算，INT8/INT4量化降低计算量，使用Flash Attention加速Attention计算。"

---

**Q34: 模型如何调优推理速度？**

> "推理速度优化几招：**量化**——INT8量化速度提升1.5-2倍，INT4提升2-3倍，精度略有下降；**KV Cache**——开启后对长序列加速明显；**Batch**——增大batch提升吞吐，但单请求延迟增加；**Flash Attention**——加速Attention计算，约20-30%提速。还有tensor parallel多卡并行，针对超长序列效果明显。"

---

**Q35: 训练完的模型多大？**

> "**完整合并后约14GB**（fp16格式）。LoRA adapter单独保存只有**30-50MB**，非常轻量。部署时如果用adapter动态加载，需要同时保留base model（14GB）和adapter（50MB）。如果合并后部署，就只有一个14GB的完整模型文件。"

### 第六类：扩展问题类

---

**Q36: 了解其他微调方法吗？**

> "了解主流高效微调方法还有几种。**Adapter Tuning**：在Transformer层之间加适配器模块，参数量比LoRA略大。**Prefix Tuning**：在输入前加可学习的前缀向量。**Prompt Tuning**：只微调embedding层的prompt。**P-tuning**：用LSTM/MLP建模prompt。各种方法都在效率和效果间做权衡，LoRA因为**效果接近全量微调、训练稳定、实现简单**而最受欢迎。"

---

**Q37: LoRA 和 QLoRA 区别？**

> "QLoRA是LoRA的**量化升级版**。核心区别是对base model做了**4-bit量化**（NF4格式），大幅降低显存。LoRA需要约16GB显存，QLoRA只需约14GB，可以在更小的GPU上训练。但QLoRA因为量化有精度损失，效果通常比LoRA略差一点点。QLoRA适合显存特别受限的场景，比如只有24GB显存的3090。"

---

**Q38: 除了 LoRA 还有什么高效微调方法？**

> "主流高效微调还有四种：**Adapter Tuning**在每层加小型适配器网络；**Prefix Tuning**在输入前加可学习前缀；**Prompt Tuning**只微调embedding层的prompt；**P-tuning**用LSTM建模soft prompt。它们共同思想都是**只微调少量参数**而不是全量微调。LoRA相比这些方法的优势是：**可拔插**方便切换任务、**训练稳定**不易崩溃、**效果最好**接近全量微调。"

---

**Q39: 你了解 RLHF 吗？**

> "RLHF是**基于人类反馈的强化学习**，用于让模型对齐人类偏好。流程是：第一步SFT监督微调，用标注数据训练模型；第二步训练Reward Model，让模型学习判断回答好坏；第三步用PPO强化学习算法优化SFT模型。ChatGPT、Claude都用RLHF让回答更有帮助、更无害。RLHF复杂度高、训练不稳定，通常在SFT之后做。"

---

**Q40: SFT 和 RLHF 区别？**

> "**SFT是监督学习**，直接用标注数据教模型回答，数据格式是（问题，优质回答）对，模型学习的是模仿这些回答。**RLHF是强化学习**，训练目标不是简单模仿，而是最大化一个学到的Reward信号，更灵活但也更难训练。简单说SFT让模型'像标注员那样回答'，RLHF让模型'回答得让人类更喜欢'。通常先SFT再做RLHF。"

---

**Q41: DeepSeek 的模型架构特点？**

> "DeepSeek有两个技术特点。**DeepSeekMoE**：MoE混合专家架构，模型分成多个专家模块，每次只激活部分专家计算，节省计算量同时保持模型容量。**DeepSeekMLA**：一种新的注意力机制，优化KV cache使用，降低推理显存。DeepSeek-7B基于这些架构创新，在保持强大性能的同时优化了训练和推理效率。"

---

**Q42: 医疗场景有什么特殊要求？**

> "医疗AI有四大要求：**准确性**——诊断建议必须准确无误，错误可能危及生命；**安全性**——不能给出有害建议，需有免责声明；**可解释性**——能解释为什么给出这个建议，让医生能审核；**隐私保护**——医疗数据敏感，必须符合HIPAA等法规，数据不能泄露。医疗AI通常定位为**辅助工具**，最终决策必须由医生做出。"

## 🎯 高频问题详细回答

### Q7: LoRA 的核心原理是什么？

**回答**：

> LoRA (Low-Rank Adaptation) 的核心思想是**低秩分解**。
>
> 假设原始全连接层权重是 `W₀ ∈ ℝ^(d×k)`，传统微调会直接更新 `W₀`，参数量是 `d×k`。
>
> LoRA 的做法是：**冻结 `W₀`，新增两个低秩矩阵 `A ∈ ℝ^(r×k)` 和 `B ∈ ℝ^(d×r)`**，其中 `r << min(d,k)`。
>
> 前向计算时：`W = W₀ + BA`
>
> 训练时只更新 A 和 B，梯度只传给 A 和 B，但反向传播时 W₀ 的梯度会被设为 0（因为 frozen）。
>
> **效果**：参数量从 `d×k` 降到 `r×(d+k)`。例如 d=4096, k=4096, r=64：
>
> - 全量微调：4096×4096 = 16M 参数
> - LoRA：(64×4096) + (4096×64) = 524K 参数 (**减少 97%**)
>
> **为什么有效？** 论文证明神经网络权重通常有较低的本征秩，新任务的知识可以压缩到低秩空间中。

> **面试快速回答（背这个）**：
> "LoRA核心是低秩分解W=W₀+BA，冻结预训练权重W₀，只训练AB两个低秩矩阵。理论依据是预训练权重有低本征秩，新任务知识可压缩到低秩空间。7B模型LoRA只需训练约0.1%参数量（7M参数），显存从28GB降到16GB，但效果接近全量微调。"

---

### Q11: 梯度检查点原理？

**回答**：

> 梯度检查点 (Gradient Checkpointing) 是用**计算换显存**的技术。
>
> **标准 Transformer forward**：保存所有层的激活值，用于 backward 计算梯度。显存占用 = O(n_layers × batch × seq_len × hidden_dim)
>
> **启用梯度检查点后**：只保存每隔几层的激活值，backward 时需要**重新计算**中间层的激活值。
>
> **权衡**：显存节省约 50-70%，但训练时间增加约 20-30%。
>
> ```python
> # Transformers 开启方式
> model.gradient_checkpointing_enable()
> # 原理：model.forward = recompute(model.forward)
> ```

> **面试快速回答（背这个）**：
> "梯度检查点是**用计算换显存**的技术。普通训练会保存所有层的激活值用于反向传播，占用大量显存。开启检查点后只保存部分激活值，反向传播时需要重新计算中间层。代价是训练时间增加20-30%，但显存节省50-70%。开启很简单，一行代码`model.gradient_checkpointing_enable()`。"

---

### Q17: 什么是灾难性遗忘？

**回答**：

> 灾难性遗忘 (Catastrophic Forgetting) 是神经网络在学习新任务时，**突然忘记之前学到知识**的现象。
>
> **原因**：全量微调时更新所有参数，新任务的数据分布可能与预训练不同，导致对预训练知识的覆盖。
>
> **LoRA 如何缓解**：
>
> 1. **冻结预训练权重**：`W₀` 固定不变，只在新增的 LoRA 参数中学习新任务
> 2. **低秩更新**：只调整一小部分参数，对预训练知识的影响最小
> 3. **正则化**：可通过设置 `lora_alpha` 控制新知识的权重
>
> **其他方法**：
>
> - Elastic Weight Consolidation (EWC)
> - Progressive Neural Networks
> - Memory Replay

> **面试快速回答（背这个）**：
> "灾难性遗忘是神经网络**学新任务时突然忘记旧知识**的现象。全量微调时所有参数都被更新，新任务数据分布可能覆盖预训练知识。LoRA通过**冻结预训练权重**来缓解——W₀固定不变，只在新增的LoRA参数中学新任务，对预训练知识影响最小。此外低秩更新，参数量少，破坏预训练知识的程度也低。"
